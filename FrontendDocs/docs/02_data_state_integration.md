# 02. Data, State & Integration

## 1. Universal Proxy for Backend Integration (BFF)

Unlike traditional setups where the frontend fetches data directly from an external third-party API, LaventeCare incorporates a **Backend-For-Frontend (BFF)** pattern. 

### The Astro API Gateway (`src/pages/api/[...path].ts`)
All external backend communication is routed through a local server endpoint mechanism:
- **Routing**: API requests hit `/api/[path]`. The proxy internally redirects them to the `PUBLIC_API_URL` (The LaventeCare Go Auth Backend).
- **Cookie Sanitization** (`cookie-utils.ts`): The proxy processes the remote server’s `Set-Cookie` directives, adjusting domain settings properly for Localhost and Edge Vercel deployments securely without hard-coded configuration loops.
- **Header Forwarding**: It seamlessly clones headers (like `X-Tenant-ID`), deletes incompatible headers (like `Host`, `Connection`), and pipes data preventing classic CORS limitations globally.

## 2. Authentication Middleware & JWKS Validation

Decoupled token validation happens at the absolute edge using **`src/middleware.ts`**. This is critical to the zero-trust nature of the LaventeCare AuthSystem.

### JWT Cryptographic Parsing
Instead of constantly hitting a `/me` endpoint on the Go Backend to check user status, the Astro Server natively decodes the incoming `access_token` JWT via `jose`:
1. **JWKS Fetching**: The middleware pulls public keys dynamically from the AuthSystem via the `/.well-known/jwks.json` endpoint.
2. **"Fail Secure" Strategy**: If the JWT signature is invalid, expired, or the JWKS schema is unreachable, the token returns `null` entirely masking the internal error state and gracefully denying permission.
3. **RBAC Payload**: The decrypted payload exposes a `role` claim. This is pushed instantly into `Astro.locals.isAdmin`.

## 3. Advanced API Client (`api-client.ts`)

Inside `src/lib/api-client.ts`, an enterprise-grade `ApiClient` class wraps the standard `fetch` API. It acts perfectly in tandem with the BFF Proxy mechanism to provide fluid client experiences.

### Environment Binding
The client directly relies on `PUBLIC_TENANT_ID` (identifying which tenant the backend must serve, e.g., `e3253710-d965-42d2-bdf8-4cf3762381c2`).

### Automated Token Refresh Queue
When a short-lived `JWT` expires, the backend will return a `401 Unauthorized` block. The `ApiClient`:
1. **Intercepts** the `401` error.
2. Checks if `isRefreshing` is currently true. If it is already refreshing, it places the current request into a **Wait Queue** (`this.refreshQueue`).
3. Makes an absolute call to `/api/v1/auth/refresh` parsing the `HttpOnly` refresh token.
4. **If Success**: Executes all stalled requests queued in memory seamlessly. The user experiences zero interruption.
5. **If Failure**: Automatically redirects to `/login`.

### CSRF Injection
For mutating actions (POST/DELETE), `ApiClient` programmatically queries `document.cookie` to extract the `csrf_token` and forcibly injects it as an `X-CSRF-Token` header.

## 4. Global State Sharing (Nanostores)

Because the application is built on an Islands Architecture, traditional React Context cannot cross the boundary between separate React islands or between Astro layouts and React modules.

To solve this, we use **Nanostores** (`nanostores`, `@nanostores/react`) located in `src/lib/stores.ts`.

### Why Nanostores?
- **Framework Agnostic**: Can be manipulated in Vanilla JS, Astro (`<script>`), and React.
- **Extremely Minimal**: Small footprint, straightforward publish-subscribe API.
- **Persistent Memory**: Often wrapped with `@nanostores/persistent` to serialize values like user session preferences smoothly into `localStorage`.

## 5. Form Management

We utilize **Zod** schema validations combined with `@conform-to/react` to parse and secure interactions generated by React Islands (like `ContactForm.tsx` or `EmailConfig.tsx`). This allows a single source of truth for validation requirements shared perfectly between our API boundaries and React.
